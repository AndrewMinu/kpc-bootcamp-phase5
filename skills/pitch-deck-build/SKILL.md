# 발표 덱 만들기 (pitch-deck-build)

## 목적
팀의 솔루션을 **심사위원 앞에서 이기는 발표**로 만든다. 산출물: `pitch-deck.html`(발표 슬라이드) + **스피커노트**(발표 대본) + **리허설 가이드**. prototype-build가 만든 `prototype.html` 데모와 이어지게 한다.

## 시작 전
- `context.md`의 `주제`·`지역`·`데이터 검증 요약`·`아이디어`·`산출물`(prototype.html)을 읽는다.
- 슬라이드 디자인은 **`prototype-craft`의 피칭덱 크래프트**(1슬라이드 1메시지·일관 골격·데이터 슬라이드 차트)를 따른다.

## 먼저 묻기 (퍼실리테이션)
- *"발표에서 심사위원이 딱 하나만 기억한다면 뭐였으면 좋겠어요?"* (핵심 메시지)
- *"누가 발표해요? 데모는 누가 돌려요?"* (역할 분담)
- *"우리 발표에서 제일 세게 밀 장면은?"* (데이터? 데모? 임팩트?)
답을 스토리라인에 반영. 팀이 정하게 한다.

## 슬라이드 구성 (피칭 4요소 = 문제근거·사용자가치·AI활용·데모)
표준 7장(팀 상황 맞게 가감):
1. **표지** — 팀·제목·한 줄 소개
2. **문제** — 누가·언제·왜 불편한가 (사용자 시나리오 한 컷)
3. **데이터 근거** — 4단계 검증 수치(비교·순위, **차트 1개**). "왜 이 문제가 진짜인지"
4. **솔루션** — 한 방(차별점) + MVP 흐름
5. **데모** — 프로토타입 실제 화면(흐름 스크린 또는 "▶ prototype.html 실행"). 데모 시나리오 1개
6. **AI 활용** — 어디서 AI가 값을 더했나
7. **임팩트·다음 단계** — 지역/사용자에 뭐가 남나, 확장

## 발표 화면(프레젠테이션 모드) 요건
- 키보드 ←/→ 네비, 페이지 인디케이터, 상단 진행바.
- **풀스크린** 발표 가능. 발표장 네트워크 없이 뜨게 단일 파일·인메모리.
- 데이터 슬라이드는 Chart.js로 시각화, 색은 테마와 통일.

## 스피커노트 (발표 대본)
각 슬라이드에 **말할 대사 2~4줄**을 함께 만든다. 별도 파일 `speaker-notes.md` 또는 덱 하단 토글로.
- 슬라이드당 30~45초 기준(총 4~5분).
- 첫 문장(훅)·마지막 문장(임팩트)은 팀이 직접 다듬게.

## 리허설 가이드
- **시간 배분**: 문제 30s · 데이터 45s · 솔루션 45s · 데모 90s · AI 30s · 임팩트 30s (조절 가능).
- **데모 리허설**: prototype.html을 실제로 눌러보며 말 맞추기. 안 되는 경우 대비(스크린 캡처 백업).
- **예상 질문(Q&A)** 3개를 팀이 뽑아 답 준비: "데이터 근거는?", "왜 이 지역?", "실제로 되나요?"

## 심화 (시간 있으면 — 2박3일 깊이)
- **모의 발표 2회 이상**: 타 팀·FT 앞에서 시간 재며 → 피드백 반영.
- **Q&A 정교화**: 예상 질문 5개로 늘리고 답을 데이터로 뒷받침.
- **훅·클로징 다듬기**: 첫 15초와 마지막 한 문장을 여러 버전으로 써보고 고른다.

## context.md 갱신
- `산출물`에 `pitch-deck.html`(+ `speaker-notes.md`) 경로.
- 로그에 "발표 덱 vN" 기록.

## 끝맺음
- 요약: 덱 경로 + 핵심 메시지 한 줄 + 발표자.
- **다음(Meta App)**: 덱·프로토타입을 공유하고 오피스아워 예약 → 피드백 반영 반복 → 발표.

## 톤·품질
- 심사 4지표에서 점수 딸 여지가 있는지 팀과 점검.
- 슬라이드는 **말을 돕는 장치**. 글 벽 금지, 큰 수치·한 메시지.
